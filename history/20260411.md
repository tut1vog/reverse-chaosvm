## 19.3: Deep cd serialization analysis + 19.4: Fix comparison C

**Outcome**: passed â€” DEEPER ROOT CAUSE FOUND

**What was done**: Enhanced ref-inject-forensics.js with field-by-field comparison and clean cd string extraction. Fixed comparison C to strip hash artifact from cd[11]. Ran live â€” found 48 fields differ because clean cd reconstruction was flawed (skips header padding, creating alignment issues).

**Actual root cause discovered**: Chrome splits the payload body at a FIELD BOUNDARY (after cd field 10, at 133 chars), padding to 144 with spaces. Our `buildInputChunks` splits at a fixed BYTE BOUNDARY (position 144), including partial field 11 content in the header. This produces different header and cdBody chunks even with identical cd content.

**Evidence**:
- Chrome header decrypted: 144 bytes, trimmed to 133 bytes (11 bytes of space padding)
- Our header: 144 bytes of continuous payload content (no padding needed since payload > 144)
- Chrome cdBody starts with `,0,[{"codec":...` (field 11 onward)
- Our cdBody starts at byte 144 of the payload (mid-field)

**Notes**: The fix is in `token/generate-token.js` `buildInputChunks()`: split the payload body at the field boundary (position 133 for Template A) rather than at a fixed 144-byte position. This may need to be parameterized per template.
