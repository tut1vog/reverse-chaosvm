## 🚨 SMOKING GUN: `vData` Field

### The Problem
The real browser's verify POST contains **39 fields**. Our bot sends 37 but is **missing `vData`** (152 chars) — a token generated by `vm-slide.enc.js`, a ChaosVM bytecode script.

### How vData is Generated
1. The show page HTML loads 3 scripts: `tcaptcha-slide.js`, `vm-slide.enc.js`, `slide-jy.js` (jQuery)
2. `vm-slide.enc.js` is a **ChaosVM script** (same architecture as `tdc.js` — `__TENCENT_CHAOS_VM` interpreter)
3. It hooks into jQuery's `$.ajaxPrefilter` pipeline
4. When `tcaptcha-slide.js` calls `$.ajax({url: "/cap_union_new_verify", data: e})`, the prefilter intercepts the request and **injects `vData`** into the POST data
5. `vData` is likely computed from the POST parameters (a signature/hash proving the request went through the ChaosVM)

### In tcaptcha-slide.js
```javascript
// The vData block in tcaptcha-slide.js only handles IE:
if(a.isLowIE()){
  var o = window.getVData && window.getVData(n.join("&"));
  o && (e.vData = o)
}
// For Chrome, vData is added by vm-slide.enc.js via jQuery ajaxPrefilter
$.ajax({type:"POST", url:"/cap_union_new_verify", data:e, ...})
```

### Why This Causes errorCode 9
Without `vData`, the server knows the request wasn't processed through the ChaosVM slide handler. It rejects the session regardless of answer correctness. This explains why:
- Valid/empty/garbage `collect` all return errorCode 9 (Attempt 8)
- TLS fingerprint doesn't matter (Attempts 6-7)
- Resource loading order doesn't matter (Attempt 9)

## POST Body Field Order (added after Task 10.5.2)

**Still getting errorCode 9 with vData present.** Root cause: the POST body fields are in the **wrong order**.

The real browser's `queryMap()` function reads the iframe URL params in their original order. Since the iframe URL = show page URL, the field order in the POST body matches the show page URL param order exactly.

Our bot hardcodes the fields in a different order — `enableAged, grayscale, dyeid, global, clientype` are at the end instead of positions 9–13 and 24. Since vData is computed over the POST body **string**, and the server likely recomputes vData in the canonical field order, the mismatch causes vData validation failure → errorCode 9.

**Correct order** (verified from HAR — show page URL params → verify POST body):
```
Positions 1-25: queryMap base (from show page URL param order)
  aid, protocol, accver, showtype, ua, noheader, fb, aged, enableAged,
  enableDarkMode, grayscale, dyeid, clientype, sess, fwidth, sid, wxLang,
  tcScale, uid, cap_cd, rnd, prehandleLoadTime, createIframeStart, global, subsid

Positions 26-38: verify-specific fields
  cdata, ans, vsig, websig, subcapclass, pow_answer, pow_calc_time,
  collect, tlg, fpinfo, eks, nonce, vlg

Position 39: vData (appended by vm-slide hook)
```
