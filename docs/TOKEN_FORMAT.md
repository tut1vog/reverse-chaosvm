# Token Format Reference — Tencent ChaosVM (TDC)

> **Authoritative document** — supersedes `TOKEN_PIPELINE.md` and `CRYPTO_ANALYSIS.md` for token format details.
> Those earlier documents contain some outdated information (e.g., "AES-based" encryption references).
> This document reflects the fully verified, byte-identical reimplementation completed in Phase 8.

## 1. Token Overview

The TDC token is a **URL-encoded string (~4,500–5,000 characters)** containing encrypted browser fingerprint data. It is generated by `TDC.getData()` in the Tencent ChaosVM runtime (`tdc.js`) and sent to a server for bot detection / device fingerprinting.

**At a glance:**

```
┌─────────────────────────────────────────────────────────────────────┐
│                     URL-Encoded Token String                        │
│                      (~4,500–5,000 chars)                           │
├─────────────────────────────────────────────────────────────────────┤
│  header(192 b64)  hash(64 b64)  cdBody(~4032 b64)  sig(120 b64)   │
│     btoa[1]         btoa[0]        btoa[2]            btoa[3]      │
├─────────────────────────────────────────────────────────────────────┤
│  Each segment: plaintext → Modified XTEA encrypt → base64 → concat │
├─────────────────────────────────────────────────────────────────────┤
│  Payload: {"cd":[...59 fields...],"sd":{"od":...,"token":...}}     │
└─────────────────────────────────────────────────────────────────────┘
```

**Purpose:** Transmit an encrypted device fingerprint from the browser to Tencent's verification servers. The fingerprint includes hardware specs, browser APIs, canvas/audio/WebGL hashes, timing data, and session metadata — 59 collector fields total.

**Verified:** The standalone reimplementation (`token/generate-token.js`) produces **byte-identical** tokens to the live `tdc.js` VM when given the same inputs, confirmed across multiple runs (Task 8.1).

## 2. Encoding Layers

The token has 5 encoding layers, applied innermost-first during generation. To decode, peel them off outermost-first.

### Layer 5 (outermost): URL-Safe Encoding

Three character replacements applied to the raw base64 string:

| Character | Replacement | Regex |
|-----------|-------------|-------|
| `+`       | `%2B`       | `/\+/g` |
| `/`       | `%2F`       | `/\//g` |
| `=`       | `%3D`       | `/=/g` |

This is **not** `encodeURIComponent()` — only these 3 characters are replaced. Implemented in `func_177` (tdc.js) / `urlEncode()` (outer-pipeline.js).

**Example from trace:** A 4,280-char base64 string becomes 4,514 chars after URL encoding (61 `+` → `%2B`, 54 `/` → `%2F`, 2 `=` → `%3D` replacements).

### Layer 4: Segment Concatenation

Four base64 segments are concatenated in a **non-sequential order**:

```
final_base64 = btoa[1] + btoa[0] + btoa[2] + btoa[3]
             = header  + hash    + cdBody  + sig
```

**Assembly order: `btoa[1] + btoa[0] + btoa[2] + btoa[3]`**

This means the **header** comes first, followed by **hash**, then the **cdBody** (encrypted collector data), and finally the **signature** (encrypted session data). The non-sequential ordering is an anti-analysis measure — a naive parser expecting `[0,1,2,3]` order would misinterpret the segments.

### Layer 3: Base64 Encoding

Each of the 4 plaintext-then-encrypted binary segments is independently base64-encoded via `btoa()` (browser) or `Buffer.toString('base64')` (Node.js). Standard base64 alphabet (A-Z, a-z, 0-9, +, /, =).

### Layer 2: Modified XTEA Encryption

Each segment is encrypted independently (ECB mode — no chaining between segments or blocks). See [Section 4: Crypto Parameters](#4-crypto-parameters) for full details.

- **Block size**: 8 bytes (two 32-bit words, little-endian)
- **Algorithm**: Modified XTEA, 32 rounds per block
- **Mode**: ECB (each 8-byte block encrypted independently)
- **Padding**: Short inputs are zero-padded to the next 8-byte boundary (via `charCodeAt()` returning `NaN` for out-of-bounds → `NaN | 0 = 0`)

### Layer 1 (innermost): Payload Assembly

The plaintext payload is a **JSON object** split across two chunks:

```json
{"cd":[...59 entries...],"sd":{"od":"C","appid":"2090803262","nonce":"0.12345678","token":"test_token_123"}}
```

This is split into:
- **cd portion**: `{"cd":[...],` (note: trailing comma, no closing brace)
- **sd portion**: `"sd":{...}}` (completes the JSON object)

The cd portion is further split into header + cdBody chunks. The sd string becomes the sig chunk. When all 4 decrypted chunks are concatenated (in input order), they reconstruct the complete JSON object.

**Important**: The cd string is built via **hand-rolled JSON concatenation** (`func_276`), NOT via `JSON.stringify()`. This is an anti-hooking measure — hooking `JSON.stringify` won't intercept the cd data.

## 3. Segment Layout

The token consists of 4 segments, each independently encrypted and base64-encoded:

```
Token byte layout (before base64/URL encoding):
┌──────────┬──────────┬────────────────────────────┬──────────┐
│  hash    │  header  │         cdBody             │   sig    │
│  48 B    │  144 B   │      ~3024 B (var)         │  ~88 B   │
│ chunk[0] │ chunk[1] │        chunk[2]            │ chunk[3] │
└──────────┴──────────┴────────────────────────────┴──────────┘

Assembly order (base64 segments):
┌──────────┬──────────┬────────────────────────────┬──────────┐
│  header  │   hash   │         cdBody             │   sig    │
│ btoa[1]  │ btoa[0]  │        btoa[2]             │ btoa[3]  │
│  192 ch  │  64 ch   │       ~4032 ch             │  120 ch  │
└──────────┴──────────┴────────────────────────────┴──────────┘
```

### Segment 0: Hash (Session Metadata)

| Property | Value |
|----------|-------|
| **Name** | hash |
| **Role** | Session metadata (timestamps, counters) |
| **Input size** | 48 bytes (fixed) |
| **Base64 size** | 64 characters (fixed) |
| **Deterministic?** | No — contains `Date.now()` timestamp |
| **Chunk index** | 0 (encrypted as chunk[0], placed at position 2 in assembly) |

**Contents:** A JSON array with session metadata, padded with spaces to 48 bytes:

```
[[4,-1,-1,1700000000000,0,0,0,0]]                 (padded with spaces to 48 bytes)
```

Fields:
- `4` — constant (likely protocol version or segment count)
- `-1, -1` — constants (status/error placeholders)
- `1700000000000` — `Date.now()` value (millisecond Unix timestamp)
- `0, 0, 0, 0` — counters/flags (always 0 in observed traces)

### Segment 1: Header (Payload Prefix)

| Property | Value |
|----------|-------|
| **Name** | header |
| **Role** | First 144 bytes of the cd+sd payload body |
| **Input size** | 144 bytes (fixed) |
| **Base64 size** | 192 characters (fixed) |
| **Deterministic?** | Yes — if collector data is identical |
| **Chunk index** | 1 (encrypted as chunk[1], placed at position 1 in assembly — first) |

**Contents:** The first 144 bytes of the "payload body" string. The payload body is constructed as:

```
cdString.slice(0, -1) + ','
```

Which transforms `{"cd":[...]}` into `{"cd":[...],` — stripping the closing `}` and adding a comma so the sd string can complete the JSON object.

The header chunk is the first 144 characters of this string, space-padded if shorter (which never happens in practice since the cd string is ~3,164 chars).

### Segment 2: cdBody (Encrypted Collector Data)

| Property | Value |
|----------|-------|
| **Name** | cdBody |
| **Role** | Remainder of the cd payload after the header |
| **Input size** | ~3,024 bytes (varies with collector data) |
| **Base64 size** | ~4,032 characters (varies) |
| **Deterministic?** | Partially — most fields are stable per session, but timestamps and some async fields vary |
| **Chunk index** | 2 (encrypted as chunk[2], placed at position 3 in assembly) |

**Contents:** Characters 144 through end of the payload body string, space-padded to the next 8-byte boundary. This contains the bulk of the 59 collector field values.

### Segment 3: Sig (Session Data / Signature)

| Property | Value |
|----------|-------|
| **Name** | sig |
| **Role** | Session data completing the JSON object |
| **Input size** | ~83–88 bytes (varies with sd field values) |
| **Base64 size** | ~120 characters (varies) |
| **Deterministic?** | Yes — if session parameters are identical |
| **Chunk index** | 3 (encrypted as chunk[3], placed at position 4 in assembly — last) |

**Contents:** The sd string produced by `JSON.stringify({sd: sdObject}).substr(1, len-1)`:

```
"sd":{"od":"C","appid":"2090803262","nonce":"0.12345678","token":"test_token_123"}}
```

This completes the JSON started by the header+cdBody: when concatenated, `{"cd":[...],` + `"sd":{...}}` = `{"cd":[...],"sd":{...}}`.

### Verified Segment Sizes (from Task 8.1 comparison-report.json)

| Segment | Binary (bytes) | Base64 (chars) | Notes |
|---------|---------------|----------------|-------|
| hash    | 48            | 64             | Fixed size |
| header  | 144           | 192            | Fixed size |
| cdBody  | 3,024         | 4,032          | Varies with cd content |
| sig     | 88            | 120            | Varies with sd content |
| **Total** | **3,304**   | **4,408**      | Before URL encoding |

After URL encoding: ~4,670 characters (varies with number of `+`, `/`, `=` in the base64 output).

## 4. Crypto Parameters

### Algorithm: Modified XTEA

The encryption uses a **Modified XTEA** (eXtended Tiny Encryption Algorithm) — a Feistel network with 32 rounds per 8-byte block.

**This is NOT AES.** Earlier analysis documents incorrectly characterized it as "AES-based." The algorithm was conclusively identified as Modified XTEA in Task 7.4, verified against 802 cipher round iterations.

### Key

Constant 128-bit key (4 × 32-bit words):

```javascript
STATE_A = [0x6257584f, 0x462a4564, 0x636a5062, 0x6d644140]
```

In bytes (little-endian): `4f 58 57 62 64 45 2a 46 62 50 6a 63 40 41 64 6d`

The key is produced by a 14-step "key schedule" in `func_271` Region 1, but the schedule always produces the same constant output — it exists purely as obfuscation.

### Delta

```javascript
DELTA = 0x9E3779B9  // Standard XTEA golden ratio constant: ⌊2^32 / φ⌋
```

### Rounds

**32 rounds** per block. The loop terminates when `sum === TARGET_SUM`:

```javascript
TARGET_SUM = 32 * DELTA = 84941944608
```

### Key Modifications

Two key-index-dependent constant additions (the "modification" in Modified XTEA):

| Key Index | Modification | Hex |
|-----------|-------------|-----|
| 1         | `+2368517`  | `+0x242405` |
| 3         | `+592130`   | `+0x090902` |
| 0, 2      | None (key value used as-is) | — |

These are applied dynamically during each half-round when selecting the key word.

### Sum Behavior (JS Semantics Critical)

The sum accumulator is **NOT truncated to 32 bits** between rounds. After 32 rounds, `sum = 84,941,944,608` — this exceeds the 32-bit range but remains in JavaScript's safe integer range (`Number.MAX_SAFE_INTEGER = 2^53 - 1`).

However, the bitwise operators (`<<`, `>>>`, `^`, `&`) used on v0/v1 **do** truncate their operands to 32-bit integers per the JS spec. This means:
- The `sum` counter stays large (for the termination check `sum !== TARGET_SUM`)
- The `sum & 3` and `(sum >>> 11) & 3` key index calculations truncate to 32-bit first
- The `v0`/`v1` values may exceed 32-bit range between rounds (the serializer extracts low 32 bits via `& 0xFF`)

### Block Processing

```
Input:  binary string (arbitrary length)
        ↓
Pad to next multiple of 8 bytes (zero-padding)
        ↓
For each 8-byte block:
  ┌─────────────────────────────────────────┐
  │  bytes[0:4] → word0 (LE)               │
  │  bytes[4:8] → word1 (LE)               │
  │  cipherRound([word0, word1])            │
  │  word0 → bytes[0:4] (LE)               │
  │  word1 → bytes[4:8] (LE)               │
  └─────────────────────────────────────────┘
        ↓
Output: encrypted binary string (same length as padded input)
```

### Cipher Round Pseudocode

```javascript
function cipherRound(v0, v1, key) {
  let sum = 0;
  const delta = 0x9E3779B9;
  const targetSum = 32 * delta; // 84941944608

  while (sum !== targetSum) {
    // v0 half-round
    idx0 = sum & 3;
    k0 = key[idx0];
    if (idx0 === 1) k0 += 2368517;
    if (idx0 === 3) k0 += 592130;
    v0 += (((v1 << 4) ^ (v1 >>> 5)) + v1) ^ (sum + k0);

    sum += delta; // NO 32-bit truncation

    // v1 half-round
    idx1 = (sum >>> 11) & 3;
    k1 = key[idx1];
    if (idx1 === 1) k1 += 2368517;
    if (idx1 === 3) k1 += 592130;
    v1 += (((v0 << 4) ^ (v0 >>> 5)) + v0) ^ (sum + k1);
  }

  return [v0, v1];
}
```

### ECB Mode

All 4 segments are encrypted independently. Within each segment, each 8-byte block is encrypted independently (no cipher block chaining). This is pure ECB mode.

### 5.2 Session Data (sd object)

The `sd` object contains 4 session parameters:

| Key | Type | Description | Example |
|-----|------|-------------|---------|
| `od` | string | Operation descriptor / endpoint code | `"C"` |
| `appid` | string | Application ID (Tencent service identifier) | `"2090803262"` |
| `nonce` | string | Client-generated nonce (typically `Math.random().toString()`) | `"0.12345678"` |
| `token` | string | Server-provided session token | `"test_token_123"` |

**Serialization:**

```javascript
// Step 1: Wrap in outer object and stringify
JSON.stringify({ sd: sdObject })
// → '{"sd":{"od":"C","appid":"2090803262","nonce":"0.12345678","token":"test_token_123"}}'

// Step 2: Strip leading '{'
result.substr(1, result.length - 1)
// → '"sd":{"od":"C","appid":"2090803262","nonce":"0.12345678","token":"test_token_123"}}'
```

The `substr(1, len-1)` removes the leading `{` so that when prepended by the cd string (`{"cd":[...],`), the result is valid JSON: `{"cd":[...],"sd":{...}}`.

### 5.3 Hand-Rolled JSON Assembly

The cd string is built by `func_276` using manual string concatenation:

```javascript
result = '{"cd":[';
for (let i = 0; i < entries.length; i++) {
  if (i > 0) result += ',';
  if (entry === null)          result += 'null';
  else if (typeof entry === 'string')  result += JSON.stringify(entry);  // quoted + escaped
  else if (typeof entry === 'number')  result += JSON.stringify(entry);  // raw number
  else if (Array.isArray(entry))       result += JSON.stringify(entry);  // standard notation
  else if (typeof entry === 'object')  result += JSON.stringify(entry);  // standard notation
}
result += ']}';
```

The sd string uses `JSON.stringify()` — only the cd string uses hand-rolled serialization.

## 7. Anti-Hooking Techniques

The ChaosVM employs multiple anti-analysis measures in the token generation pipeline:

### 7.1 Hand-Rolled JSON Serialization

The cd array is serialized using manual string concatenation (`func_276`) rather than `JSON.stringify()`. This means:
- Hooking `JSON.stringify` does **not** intercept the cd payload
- The serialization logic is embedded in VM bytecode, making it harder to find
- Only the sd string uses `JSON.stringify` (a deliberate inconsistency)

### 7.2 Internal Crypto Dispatch

The encryption function (`func_271`) is called through the VM's internal function dispatch mechanism, not through any `window`-exposed API. There is no `ChallengeEncrypt` function visible on `window` — it's resolved internally via bytecode function references.

### 7.3 Custom PRNG / Nonce Generation

Nonce values and some fingerprint entropy are derived from `Math.random()` and `performance.now()` in ways that are difficult to predict or replay without controlling the exact browser state.

### 7.4 Self-Modifying Bytecode

At PC 40178, a `THROW` opcode is dynamically patched to become a `CJMP` (conditional jump) during execution. This defeats static disassembly — a disassembler that doesn't account for runtime bytecode modification will see a `THROW` and assume the code path is unreachable.

### 7.5 Obfuscated Key Schedule

The 14-step key schedule in `func_271` Region 1 appears complex (multiple function calls, intermediate states) but always produces the same constant 4-word key. This wastes an analyst's time analyzing a "key derivation" that is actually just obfuscated constant loading.

### 7.6 Webpack Module Loading

The 59 collector functions are loaded via a webpack-style module system (`func_0` / `func_99`), with module indices that don't correspond to cd array indices. This indirection makes it harder to map "module 42 → timezone offset" without running the code.

### 7.7 Dual Timestamp Validation

Multiple timestamp fields (cd[16], cd[52], cd[53], hash timestamp) provide cross-validation. The server can detect:
- Replayed tokens (timestamps too old)
- Manufactured tokens (timestamps inconsistent with each other)
- Time-frozen environments (all timestamps identical)

## 9. How to Generate a Token

### Using the CLI

```bash
# Generate with default Chrome/Linux profile
node token/cli.js

# Generate with a specific profile
node token/cli.js --profile headless-chrome

# Generate with custom session data
node token/cli.js --appid 2090803262 --nonce 0.12345678 --token my_token

# Generate with frozen timestamp (for reproducibility)
node token/cli.js --timestamp 1700000000000
```

### Programmatic Usage

```javascript
const { generateToken } = require('./token/generate-token.js');
const { buildDefaultCdArray } = require('./token/collector-schema.js');

// 1. Build collector data (59 fields)
const cdEntries = buildDefaultCdArray({
  platform: 'Linux x86_64',
  userAgent: 'Mozilla/5.0 (X11; Linux x86_64) ...',
  screenWidth: 1920,
  screenHeight: 1080,
  // ... other profile values
});

// 2. Define session data
const sdObject = {
  od: 'C',
  appid: '2090803262',
  nonce: '0.12345678',
  token: 'your_session_token',
};

// 3. Generate token
const token = generateToken(cdEntries, sdObject, Date.now());
// → "V5OYLxrEF0DY%2B%2B7Kr1A07k84OCzJSmXcx0%2BoUZy..." (~4,670 chars)
```

### Step-by-Step Pipeline

For maximum control, you can invoke each pipeline stage individually:

```javascript
const { buildCdString, buildSdString, urlEncode } = require('./token/outer-pipeline.js');
const { encryptSegments } = require('./token/crypto-core.js');
const { buildInputChunks } = require('./token/generate-token.js');

// Step 1: Serialize collector data
const cdString = buildCdString(cdEntries);    // '{"cd":[1,"linux",2,800,...]}'

// Step 2: Serialize session data
const sdString = buildSdString(sdObject);     // '"sd":{"od":"C","appid":"2090803262",...}}'

// Step 3: Build input chunks
const chunks = buildInputChunks(cdString, sdString, 1700000000000);
// chunks[0] = hash (48 bytes)
// chunks[1] = header (144 bytes)
// chunks[2] = cdBody (~3024 bytes)
// chunks[3] = sig (~83 bytes)

// Step 4: Encrypt + base64
const btoaSegments = encryptSegments(chunks);
// btoaSegments[0] = hash base64 (64 chars)
// btoaSegments[1] = header base64 (192 chars)
// btoaSegments[2] = cdBody base64 (~4032 chars)
// btoaSegments[3] = sig base64 (120 chars)

// Step 5: Assemble in order [1, 0, 2, 3]
const assembled = btoaSegments[1] + btoaSegments[0] + btoaSegments[2] + btoaSegments[3];

// Step 6: URL-encode
const token = urlEncode(assembled);
```

## Appendix B: Decoding a Token

To reverse the encoding and inspect a token:

```javascript
const { decryptSegments } = require('./token/crypto-core.js');

// 1. URL-decode
let raw = token.replace(/%2B/g, '+').replace(/%2F/g, '/').replace(/%3D/g, '=');

// 2. Split segments (header=192, hash=64, cdBody=variable, sig=remainder)
const header_b64 = raw.substring(0, 192);
const hash_b64   = raw.substring(192, 192 + 64);
const sig_b64    = raw.substring(raw.length - 120);
const cdBody_b64 = raw.substring(192 + 64, raw.length - 120);

// 3. Decrypt each segment
const [hash, header, cdBody, sig] = decryptSegments([hash_b64, header_b64, cdBody_b64, sig_b64]);

// 4. Reconstruct payload
const payload = header.trimEnd() + cdBody.trimEnd() + sig.trimEnd();
// → '{"cd":[...],"sd":{...}}'

// 5. Parse
const data = JSON.parse(payload);
console.log(data.cd);  // 59-element array
console.log(data.sd);  // {od, appid, nonce, token}
```
